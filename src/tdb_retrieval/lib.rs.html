1//! Latent-space retrieval for `TardigradeDB` — find relevant memory cells by attention scoring.
2//!
3//! `TardigradeDB` does not use cosine similarity over text embeddings. Relevance is computed
4//! as **scaled dot-product attention** directly in the model's latent space:
5//!
6//! ```text
7//! score(q, k) = (q · k) / √d_k
8//! ```
9//!
10//! This is the same formula used inside a transformer's attention head. Querying memory
11//! with a key vector from a live inference pass means the retrieval metric is identical
12//! to how the model itself relates tokens — no separate embedding model required.
13//!
14//! # Three-Stage Retrieval Pipeline
15//!
16//! ```text
17//! query key vector
18//!       │
19//!       ▼
20//! ┌─────────────────────────────────────────┐
21//! │  PerTokenRetriever (primary)            │
22//! │  • Inverted Multi-Key Index             │
23//! │  • INT8 per-token scoring (Top5Avg)     │
24//! │  • 100% recall at 100 memories          │
25//! │  • Tuneable via PerTokenConfig           │
26//! └─────────────┬───────────────────────────┘
27//!               │ insufficient results
28//!               ▼
29//! ┌─────────────────────────────────────────┐
30//! │  BruteForceRetriever (fallback)         │
31//! │  • Full f32 exhaustive scan             │
32//! │  • Owner-filtered per-agent partitions  │
33//! │  • Valid up to ~10K cells per agent     │
34//! └─────────────────────────────────────────┘
35//!
36//! Parallel hot cache (separate from pipeline):
37//! ┌─────────────────────────────────────────┐
38//! │  SemanticLookasideBuffer (hot path)     │
39//! │  • Fixed-capacity LRU, INT8 keys        │
40//! │  • Target: < 5 μs via SIMD dot product  │
41//! │  • Warmed on every read/write           │
42//! └─────────────────────────────────────────┘
43//! ```
44//!
45//! ## Why Brute Force, Not ANN?
46//!
47//! The [MemArt paper] (which this architecture cites) demonstrates that at typical
48//! per-agent memory scales (<10K blocks), SIMD brute-force matmul outperforms HNSW
49//! and other approximate nearest-neighbor indexes. ANN indexes pay a build-time and
50//! index-memory overhead that only pays off beyond ~100K vectors. For larger cold
51//! indexes, `tdb-index` provides a Vamana graph (DiskANN-style).
52//!
53//! Additionally, HNSW is unreliable for Q/K distributions that shift over time
54//! (which happens as the model processes different contexts), because its graph
55//! edges are built against a snapshot distribution.
56//!
57//! ## Semantic Lookaside Buffer
58//!
59//! The [`SemanticLookasideBuffer`] is modeled on a CPU's Translation Lookaside Buffer:
60//! a fast fixed-size cache for the most recently used entries. It stores keys in
61//! **symmetric INT8 quantization** — one scale per full vector instead of per-group —
62//! which allows the dot product to be computed as integer arithmetic using NEON
63//! `SDOT` intrinsics (ARM) or equivalent auto-vectorized code on x86.
64//!
65//! # Usage
66//!
67//! ```rust
68//! use tdb_retrieval::attention::BruteForceRetriever;
69//!
70//! let mut retriever = BruteForceRetriever::new();
71//!
72//! // Index a set of key vectors (from stored cells).
73//! retriever.insert(/* cell_id */ 0, /* owner */ 1, /* layer */ 12, &[1.0, 0.0, 0.0, 0.0]);
74//! retriever.insert(/* cell_id */ 1, /* owner */ 1, /* layer */ 12, &[0.0, 1.0, 0.0, 0.0]);
75//!
76//! // Query with a key from the current inference pass.
77//! let results = retriever.query(&[1.0, 0.0, 0.0, 0.0], /* k */ 1, /* owner_filter */ None);
78//! assert_eq!(results[0].cell_id, 0); // exact-match cell scores highest
79//! ```
80//!
81//! [`SemanticLookasideBuffer`]: slb::SemanticLookasideBuffer
82//! [MemArt paper]: https://arxiv.org/abs/2409.17264
