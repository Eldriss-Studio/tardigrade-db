Storage Layer — How fast can we persist and read KV tensors?
Q4 quantize — compress f32 → 4-bit (GGML Q4_0)
Measures how fast we compress a raw f32 activation vector into 4-bit quantized form for storage. This runs on every mem_write — it's the cost of shrinking a KV tensor to ~4x smaller before hitting disk. Lower is better. Scales linearly with vector dimension.
Varies: 64, 128, 256, 512, 1024 floats
Q4 dequantize — decompress 4-bit → f32
The reverse of quantization: expand a stored 4-bit tensor back to full f32. This runs on every mem_read when a cell is loaded from disk. If this is slow, cold-path reads bottleneck here rather than in retrieval.
Varies: 64, 128, 256, 512, 1024 floats
Q4 round-trip — quantize + dequantize end-to-end
Full compression cycle: f32 → Q4 → f32. This is the total codec overhead per vector. Useful for understanding the combined cost that every write-then-read path pays.
Varies: 128, 256, 512 floats
Block pool append — Q4-compress and fsync one cell to segment
The full storage write path: serialize a MemoryCell, Q4-compress both key and value vectors, append the binary record to the active segment file, and fsync to guarantee durability. Dominated by the fsync syscall — this is why batch writes exist (one fsync for N cells).
dim=128, single cell per iteration
Block pool random read — dequantize one cell from 10K on disk
Read a single cell by ID from a pool of 10,000 stored cells. Involves seeking to the cell's offset in the segment file and Q4-dequantizing its vectors. This is the cold-path read cost — the price of going to disk when the SLB cache misses.
dim=128, 10K cells pre-loaded, pseudo-random access pattern
Retrieval Layer — How fast can we find relevant memories?
FP32 dot product — baseline attention score
Raw floating-point dot product between two vectors. This is the ground-truth speed for computing one attention score (q · k / √d_k) — the fundamental operation behind memory retrieval. This is the baseline that INT8 quantization aims to beat.
Varies: dim 64, 128, 256, 512
INT8 dot product — NEON-accelerated attention score
Same dot product but using INT8 scalar quantization with ARM NEON SIMD instructions. The speedup over FP32 (typically 5–9x) is the payoff for quantizing keys in the SLB cache. This is the inner loop of every hot-path retrieval — thousands of these per query.
Varies: dim 64, 128, 256, 512
SLB query — hot-path INT8 cache lookup (top-5)
End-to-end Semantic Lookaside Buffer query: scan all cached entries using INT8 dot products, rank by attention score, return the top 5 most relevant cells. This is the fastest possible retrieval path — recently accessed cells are served entirely from this LRU cache without touching disk or the graph index.
Varies: 256, 1024, 4096 entries · dim=128
Organization Layer — How fast can we index and recover?
Vamana index build — DiskANN-style graph construction
Build the approximate nearest neighbor graph from scratch. This is an offline operation that runs when the cell count crosses a threshold (~10K). The graph enables sub-linear query time — without it, every query scans all cells. Build cost is O(n²) for medoid selection plus O(n × beam_width) for edge wiring.
Varies: 100, 500, 1000 nodes · dim=32 · max_degree=16
Vamana query — greedy beam search over 1K-node graph
A single approximate nearest neighbor query: start at the medoid, greedily expand the best candidates through graph edges, return the top 10 results. This is the cold-path retrieval for agents with large memory pools where brute-force scanning is too slow.
1K nodes · dim=32 · top-10
WAL append — fsync'd causal edge writes
Write one causal relationship ("cell A caused cell B") to the write-ahead log with fsync durability. Every mem_write with a parent_cell_id pays this cost. The WAL ensures the causal graph survives crashes — if the engine dies mid-write, the edge is recovered on restart.
Varies: 100, 1000 edges per batch
WAL replay — crash recovery: read 1K edges from disk
On engine startup, the WAL is replayed to reconstruct the causal graph. This measures how long it takes to read 1,000 edge records from the log and rebuild the in-memory TraceGraph. Faster replay = faster crash recovery = less downtime.
1K edges
Engine — End-to-end: what does a real operation cost?
Engine mem_write — single cell persist with fsync
The full write pipeline that an inference hook calls: assign a cell ID → build the MemoryCell → Q4-compress key and value → append to segment with fsync → insert into the retriever index → update governance scores. This is the end-to-end cost of "remember this KV pair." Dominated by fsync — batch writes amortize it.
dim=64 · single cell
Engine mem_read — full pipeline: SLB → retriever → governance
The full read pipeline that an inference hook calls: check the SLB cache → fall back to brute-force retriever → apply recency decay → boost governance scores → return the top 5 most relevant cells. This is the end-to-end cost of "what do I remember about this query?" — the number that determines whether memory retrieval fits within an inference budget.
1K cells · dim=64 · top-5
