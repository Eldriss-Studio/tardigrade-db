Benchmark V1 Narrative

Transparent, comparable memory benchmark results for TardigradeDB vs external systems. This page explains what the files mean and how to interpret them safely.

Latest files:
  • Latest full report (Markdown)
  • Latest full run (JSON raw data)
  • Latest full comparison vs previous baseline
Per-run snapshots: dev/bench-v1/runs/<run_id>/ in GitHub Pages.

What This Benchmark Measures

  • Quality: correctness score over dataset items.
  • Reliability: how many items are ok, skipped, or failed.
  • Fairness: shared evaluator config (judge/answerer/top-k/prompts) across systems per run.
  • Transparency: raw per-item evidence, judgments, and manifest metadata are retained in JSON.

How To Interpret run_validity

Status Meaning How to use result
comparable All systems produced successful outputs for the evaluated items. Safe to compare score/latency head-to-head.
degraded At least one system had partial skips/failures. Use with caution; include reliability caveat in any claim.
invalid At least one system had no successful results. Do not claim quality winner; treat as infra or compatibility failure.

CI Publication Model

  • PR CI: smoke profile (10-minute cap) with quality gates and artifact upload.
  • Scheduled benchmark workflow: full matrix artifacts + GitHub Pages publish.
  • Quality gate policy: fail on run_validity=invalid or high non-ok ratio.

Known Limits

  • When no API key is available, evaluator falls back to deterministic lexical scoring.
  • External services may be unavailable; those outcomes are explicit (skipped/failed), never dropped.
