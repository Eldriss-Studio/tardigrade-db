Expand description

Per-token retriever using Inverted Multi-Key Index with SoA layout.

Multiple token entries point back to the same CellId. On query, each query token scores against all stored tokens. Aggregation depends on ScoringMode. Token vectors are stored contiguously in a flat arena for cache-friendly sequential access.

How to aggregate per-token scores into a cell score.

Tuneable retrieval thresholds (Parameter Object pattern).

§last_scored_cell_count: usize

Number of distinct cells fully reranked during the previous query.

§corpus_sum: Vec<f32>

Running sum of stored f32 token vectors, per dimension. Used by mean-centered refinement to subtract the corpus’s shared high-energy direction. Updated on insert; not decremented on remove (the bias is small in practice and is fully refreshed when the engine rebuilds the retriever by replaying inserts from durable storage).

§corpus_token_count: usize

Number of f32 token vectors that have contributed to corpus_sum.

§corpus_sq_sum: Vec<f32>

Running sum of outer products (x_j * x_k) across all tokens, stored as a flat dim*dim matrix in row-major order. Used to compute corpus covariance for ZCA whitening.

§whitening_cache: Option<Vec<f32>>

Cached ZCA whitening matrix (dim*dim, row-major). Invalidated when corpus_token_count changes.

§whitening_cache_count: usize

Token count at which whitening_cache was last computed.

§token_cache: Mutex<LruCache<CellId, Arc<Vec<f32>>>>

Bounded cache of decoded f32 token matrices, keyed by CellId. Used by query_with_source to amortize Q4-decode cost over hot candidates. Cache-Aside semantics — the source remains the source of truth; the cache only fills as queries decode cells.

§int8_tokens: Mutex<LruCache<CellId, Arc<Vec<QuantizedToken>>>>

In-memory INT8-quantized retrieval tier — the scoring tier.

Bounded LRU of Vec<QuantizedToken> per cell at INT8 precision (~150 KB per Qwen-0.6B-shaped cell). Read by the lazy query_with_source path before falling back to the Q4-archival source. Populated at Retriever::insert time when lazy mode is active; LRU eviction kicks in once the cache reaches its configured capacity.

Cold-cell behavior: when a cell is evicted (or never seen, e.g. after engine restart before the tier is re-populated), the lazy retriever falls back to reading the Q4 archival source. Q4 precision is coarser; recall on cold cells degrades. The LRU capacity should be sized to comfortably hold the active working set so this fallback is rare in steady state.

§populate_token_store: bool

Whether insert populates the in-memory TokenStore.

true (default): legacy eager mode — query and dequantized_tokens_for_cell work against the in-memory store. Memory cost is ~130 KB per cell at Qwen3-0.6B-shaped tokens.

false: lazy mode — only metadata + cell_summaries are kept per cell. query returns empty (callers must use query_with_source with a CellSource). Memory cost is ~4 KB per cell.

The engine constructs the retriever in lazy mode; tests that need the eager API construct it in default mode.

pub fn dequantized_tokens_for_cell( &self, cell_id: CellId, ) -> impl Iterator<Item = (Vec<f32>, OwnerId)> + '_

Iterate stored f32-dequantized tokens for a single cell, paired with owner.

Yields (token_f32, owner) for each token of cell_id. Returns an empty iterator if the cell is not present. Used by refinement strategies that need to re-score a small set of candidates outside the INT8 fast path.

Eager-mode only: reads from the in-memory TokenStore. For the lazy path, use Self::tokens_for_cell_via_source which falls back to a CellSource when TokenStore is empty.

Query against a CellSource with an optional upstream candidate filter from earlier pipeline stages.

Decodes tokens for candidate cells on demand instead of relying on the in-memory TokenStore. The result set is observationally identical to Retriever::query modulo INT8 vs f32 dot-product rounding (the eager path quantizes both sides; the lazy path scores in f32). When upstream_candidates is Some, only cells in that set are scored — Phase-1 selection still runs and the two sets are intersected.

fn load_decoded_tokens( &self, cell_id: CellId, source: &dyn CellSource, expected_dim: usize, ) -> Option<Arc<Vec<f32>>>

Cache-Aside: cache hit → return; miss → load via source, decode, insert into cache, return. The cache stores decoded f32 token data wrapped in Arc so hits are cheap to clone.

In lazy mode, the INT8 retrieval tier is checked first — it holds full-precision quantized tokens captured at insert time (before any Q4 round-trip). If the cell is present there, we dequantize from INT8 (precise) rather than from Q4 (coarse). The fallback to source (Q4 archival) remains for cells the retriever has not yet seen — for example, when the retriever is being rebuilt from durable storage on engine restart.
